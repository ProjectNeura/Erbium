Support guidance

These notes cover the most common join, stop, and leave issues without requiring a separate runbook.

Before you join

Set a realistic runtime window and keep the submitted credentials accessible.

  • Shorter runtime requests reduce total wait time for the rest of the queue.
  • Available GPU badges are informational; new jobs are still placed into the scheduler queue.
  • Use a memorable job name so you can identify and remove the request later if plans change.

If a stop request fails

A stop action only succeeds when the password matches the current running job.

  • Confirm a job is actively running before submitting the password.
  • Use the password associated with the currently active session.
  • Retry after refresh if the running-job panel has not loaded yet.

If a join or leave request fails

Join validates required fields first. Leave only succeeds for queued jobs with exact credentials.

  • Check that runtime is a positive number of hours before joining.
  • Use the original job name exactly as submitted when leaving.
  • If the job has already started running, it is no longer in the queue and cannot be removed here.
